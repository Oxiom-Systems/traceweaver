# Validation Planning Checklist

Use this checklist for each need or release objective under validation
planning or validation-closure judgment. Missing items do not all have the
same severity; judge them by whether the validation claim can safely stand as
acceptance evidence.

## Per Need Or Release Objective

- [ ] Need ID or source intent is linked.
- [ ] Intended use is described.
- [ ] Operating context is described.
- [ ] Stakeholder, rater, representative user, or approved proxy is named.
- [ ] Rater role is recorded.
- [ ] Rater relationship to the work is recorded.
- [ ] Scenario reflects the intended context.
- [ ] Acceptance signal is observable.
- [ ] Evidence collection method is defined.
- [ ] When verification evidence exists, it is linked and labeled as
      verification, not validation.
- [ ] Adoption evidence or approval package inputs are defined.
- [ ] Deferred validation has owner, trigger, expected evidence, and
      approval.

## Hold When

Hold the validation decision when any of these are true:

- [ ] Implementer self-assessment is the only evidence.
- [ ] Rater is unnamed.
- [ ] Scenario is not representative of intended use or operating context.
- [ ] Acceptance criteria do not describe an observable outcome in the
      intended context.
- [ ] Evidence path is missing.
- [ ] Deferred validation has no approved gap or trigger.

A held decision records which hold conditions apply and what would clear
them. Holding is the correct outcome when evidence is missing; do not lower
the gate to close the claim.
