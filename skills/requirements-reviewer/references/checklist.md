# Checklist

For each candidate requirement:

- [ ] Has stable ID.
- [ ] Has clear type.
- [ ] Has current text.
- [ ] Has parent need, source, risk, regulation, or approved decision.
- [ ] Has owner.
- [ ] Has status and approval state.
- [ ] Has rationale.
- [ ] Is necessary.
- [ ] Is singular.
- [ ] Is unambiguous.
- [ ] Is complete enough to interpret.
- [ ] Is feasible.
- [ ] Is verifiable.
- [ ] Has verification method: inspection, analysis, demonstration, or test.
- [ ] Has pass/fail or review condition.
- [ ] Has validation path to need, scenario, stakeholder, or operational signal.
- [ ] Avoids unnecessary implementation detail.
- [ ] Avoids vague words, escape clauses, and open-ended lists.
- [ ] Does not duplicate or conflict with another requirement.
- [ ] Has version/change history when changed.

## Block Approval When

- [ ] No parent/source/need exists.
- [ ] No objective verification path exists.
- [ ] Actor or obligated system is unclear.
- [ ] Outcome is subjective or immeasurable.
- [ ] Multiple obligations are combined.
- [ ] Design is prescribed without approved constraint rationale.
- [ ] Requirement conflicts with another approved item.
- [ ] Text is actually an idea, task, assumption, or design note.
